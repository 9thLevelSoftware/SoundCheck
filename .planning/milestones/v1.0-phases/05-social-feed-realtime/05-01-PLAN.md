  <action>
    **Migration 020_add-feed-read-cursors.ts:**
    Create feed_read_cursors table for tracking per-user last-read position per feed tab. Follow the exact pattern from research:
    - id: uuid PK with uuid_generate_v4() default
    - user_id: uuid NOT NULL, references users ON DELETE CASCADE
    - feed_type: varchar(20) NOT NULL (values: 'friends', 'event', 'happening_now')
    - last_seen_at: timestamptz NOT NULL
    - last_seen_checkin_id: uuid (nullable)
    - updated_at: timestamptz with CURRENT_TIMESTAMP default
    - UNIQUE constraint on (user_id, feed_type)
    Follow existing migration pattern from 019_add-user-badges-metadata.ts.

    **FeedService.ts:**
    Create new FeedService class with the following methods:

    1. `getFriendsFeed(userId, cursor?, limit=20)` - Query checkins from followed users (JOIN user_followers) with cursor-based pagination using (created_at, id) row-value comparison. Include user avatar, username, event name, venue name, photo URL, badge earned indicator, toast count, comment count, has_user_toasted. Cache result in Redis with 60s TTL using key pattern `feed:friends:{userId}:{cursor||'head'}`. Use existing getCache/setCache from cache.ts.

    2. `getEventFeed(eventId, cursor?, limit=20)` - Query all checkins for a specific event (WHERE event_id = $1). Same fields and cursor pattern. Cache with key `feed:event:{eventId}:{cursor||'head'}`, 60s TTL.

    3. `getHappeningNow(userId)` - Query friends checked in to events happening today, grouped by event. Use json_agg for friend grouping with GROUP BY event_id. Expiry strategy: COALESCE(event.end_time + INTERVAL '1 hour', event.start_time + INTERVAL '4 hours', (event.event_date + INTERVAL '1 day')::timestamptz). Cache with key `feed:happening:{userId}`, 30s TTL (shorter for live data).

    4. `getUnseenCounts(userId)` - For each feed_type in feed_read_cursors, count checkins newer than last_seen_at. Return { friends: N, event: N, happening_now: N }.

    5. `markFeedRead(userId, feedType, lastSeenAt, lastSeenCheckinId)` - UPSERT feed_read_cursors row for this user+feedType.

    6. `invalidateUserFeedCache(userId)` - Delete feed cache keys for this user using cache.delPattern(`feed:friends:${userId}:*`). Also delete happening_now cache.

    7. `invalidateEventFeedCache(eventId)` - Delete feed cache keys for this event using cache.delPattern(`feed:event:${eventId}:*`).

    Helper functions (module-level exports, not class methods):
    - `encodeCursor({ createdAt, id })` - Base64url encode JSON cursor
    - `decodeCursor(encoded)` - Base64url decode, return { createdAt, id } or null

    Use Database.getInstance() for DB access (existing pattern). Use getCache/setCache/deleteCache from cache.ts. Use CacheTTL.SHORT (60s) for feed cache.

    The friends feed query MUST include:
    - c.id, c.user_id, c.event_id, c.created_at, c.photo_url
    - u.username, u.profile_image_url (as userAvatarUrl)
    - e.event_name (as eventName), v.name (as venueName)
    - COUNT(DISTINCT t.id) as toastCount
    - COUNT(DISTINCT cm.id) as commentCount
    - EXISTS check-in has badge in user_badges as hasBadgeEarned
    - EXISTS user toasted as hasUserToasted
    - Standard ORDER BY c.created_at DESC, c.id DESC

    Map results to a FeedItem interface:
    ```typescript
    interface FeedItem {
      id: string;
      checkinId: string;
      userId: string;
      username: string;
      userAvatarUrl: string | null;
      eventId: string;
      eventName: string;
      venueName: string;
      photoUrl: string | null;
      createdAt: string;
      hasBadgeEarned: boolean;
      toastCount: number;
      commentCount: number;
      hasUserToasted: boolean;
    }

    interface FeedPage {
      items: FeedItem[];
      nextCursor: string | null;
      hasMore: boolean;
    }

    interface HappeningNowGroup {
      eventId: string;
      eventName: string;
      venueName: string;
      friends: { userId: string; username: string; profileImageUrl: string | null }[];
      totalFriendCount: number;
      lastCheckinAt: string;
    }
    ```

    **FeedController.ts:**
    Create FeedController class with the following endpoint handlers. Each handler extracts userId from req.user (set by authenticateToken middleware) and validates query params.

    1. `getFriendsFeed(req, res)` - GET /api/feed/friends?cursor=X&limit=N
       - Validate limit (1-50, default 20), cursor is optional string
       - Call feedService.getFriendsFeed(userId, cursor, limit)
       - Return { success: true, data: result }

    2. `getEventFeed(req, res)` - GET /api/feed/events/:eventId?cursor=X&limit=N
       - Validate eventId is present, limit (1-50, default 20)
       - Call feedService.getEventFeed(eventId, cursor, limit)
       - Return { success: true, data: result }

    3. `getHappeningNow(req, res)` - GET /api/feed/happening-now
       - Call feedService.getHappeningNow(userId)
       - Return { success: true, data: result }

    4. `getUnseenCounts(req, res)` - GET /api/feed/unseen
       - Call feedService.getUnseenCounts(userId)
       - Return { success: true, data: result }

    5. `markRead(req, res)` - POST /api/feed/mark-read
       - Body: { feedType: 'friends'|'event'|'happening_now', lastSeenAt: ISO string, lastSeenCheckinId?: string }
       - Validate feedType is one of the three valid values
       - Call feedService.markFeedRead(userId, feedType, lastSeenAt, lastSeenCheckinId)
       - Return { success: true }

    All handlers wrapped in try/catch returning { success: false, error: message } on failure.

    **feedRoutes.ts:**
    Replace the existing single-route file. Keep authenticateToken middleware. Add all five routes:
    - GET /friends -> feedController.getFriendsFeed
    - GET /events/:eventId -> feedController.getEventFeed
    - GET /happening-now -> feedController.getHappeningNow
    - GET /unseen -> feedController.getUnseenCounts
    - POST /mark-read -> feedController.markRead

    Keep backward compatibility: the old GET / route still works by forwarding to getFriendsFeed (so existing mobile app doesn't break until updated).

    **CheckinService.ts - cache invalidation hook:**
    In the `createEventCheckin` method, AFTER the check-in is successfully created and AFTER the badge eval job is enqueued, add cache invalidation:
    1. Import FeedService or use cache.delPattern directly
    2. Get follower IDs of the check-in creator: `SELECT follower_id FROM user_followers WHERE following_id = $1`
    3. For each follower, call `cache.delPattern('feed:friends:' + followerId + ':*')` to invalidate their friends feed cache
    4. Invalidate the event feed cache: `cache.delPattern('feed:event:' + eventId + ':*')`
    5. Invalidate happening now cache for each follower: `cache.del('feed:happening:' + followerId)`
    6. Also invalidate the creator's own feed cache (they might see their own check-in in event feed)
    7. This is fire-and-forget (don't await, don't let errors block check-in response). Wrap in try/catch, log on error.
    8. Also add this same hook to the legacy `createCheckin` method for backward compat.

    IMPORTANT: Do NOT modify the check-in creation logic itself. Only ADD cache invalidation after the existing flow completes.
