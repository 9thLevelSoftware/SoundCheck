  <files>

  <action>
    Create `backend/src/services/EventSyncService.ts` -- the sync pipeline orchestrator.

    Class uses Database.getInstance(), TicketmasterAdapter, and BandMatcher.

    **Core method: `runSync(regionId?: string)`**
    1. Create event_sync_log entry with status='running'.
    2. Load active sync regions from `sync_regions` table (or single region if regionId provided). If no regions exist, log warning and return (pipeline is a no-op until regions are configured).
    3. For each region:
       a. Build date window: today to today+30 days (ISO 8601 format for TM API).
       b. Call `ticketmasterAdapter.fetchAllEventsForRegion(latlong, radius, startDate, endDate)` to get NormalizedEvent[].
       c. For each NormalizedEvent:
          - Call `bandMatcher.matchOrCreateVenue(event.venue)` to resolve venue_id.
          - For each attraction in event.attractions, call `bandMatcher.matchOrCreateBand(attraction.name, attraction.externalId, attraction.genre, attraction.imageUrl)` to resolve band_ids.
          - UPSERT event: `INSERT INTO events (venue_id, event_date, event_name, start_time, ticket_url, ticket_price_min, ticket_price_max, source, external_id, is_verified, status) VALUES (...) ON CONFLICT (source, external_id) DO UPDATE SET event_name = EXCLUDED.event_name, ticket_url = EXCLUDED.ticket_url, ticket_price_min = EXCLUDED.ticket_price_min, ticket_price_max = EXCLUDED.ticket_price_max, status = EXCLUDED.status, updated_at = CURRENT_TIMESTAMP RETURNING id, (xmax = 0) AS is_new`.
          - Track whether INSERT or UPDATE via `xmax = 0` check.
          - Upsert event_lineup entries for each matched band: `INSERT INTO event_lineup (event_id, band_id, set_order, is_headliner) VALUES ($1, $2, $3, $4) ON CONFLICT (event_id, band_id) DO NOTHING`.
          - If status changed from previous value (detect by checking old status before upsert), call `handleStatusChange(eventId, newStatus, oldStatus)` to notify affected users.
          - Increment counters (events_created, events_updated, events_skipped, bands_created, bands_matched, venues_created).
       d. Update region's `last_synced_at`.
    4. Update event_sync_log entry with status='completed', counters, completed_at.
    5. On error: update event_sync_log with status='failed', error_message, completed_at.

    **Helper method: `handleStatusChange(eventId, newStatus, oldStatus)`**
    - If newStatus !== oldStatus AND (newStatus is 'cancelled' or 'rescheduled'):
      - Query users who checked in to this event: `SELECT DISTINCT user_id FROM checkins WHERE event_id = $1`.
      - For each user, create notification via direct INSERT into notifications table (follow NotificationService pattern from Phase 1).

    **Helper method: `ingestSingleEvent(tmEvent: TicketmasterEvent)`**
    - For on-demand lookups (Plan 03). Normalizes a single TM event, resolves venue+bands, upserts event+lineup.
    - Returns the event object or null if event has no venue.

    **Graceful degradation:**
    - If TICKETMASTER_API_KEY is not set, constructor logs info message and sets a flag. runSync() returns immediately with "API key not configured" log.
    - All database operations use try/catch per-event (one bad event doesn't kill the whole sync).
    - Wrap the entire region loop in try/catch so one failed region doesn't stop others.

             removeOnFail: { count: 200 },

       - Guard: If REDIS_URL is not set, export a null queue and log warning. All consumers must check for null before using.

    3. Create `backend/src/jobs/eventSyncWorker.ts`:
       - Import { Worker } from 'bullmq'.
       - Import createBullMQConnection from '../config/redis'.
       - Import EventSyncService.
       - `startEventSyncWorker()`:
         - If REDIS_URL not set, log warning and return null.
         - Create Worker for queue 'event-sync':
           ```
           const worker = new Worker('event-sync', async (job) => {
             const syncService = new EventSyncService();
             if (job.name === 'scheduled-sync') {
               await syncService.runSync();
             } else if (job.name === 'check-cancellations') {
               await syncService.runSync(); // Same flow, TM API returns current status
             } else if (job.name === 'region-sync') {
               await syncService.runSync(job.data.regionId);
             }
           }, {
             connection: createBullMQConnection(),
             concurrency: 1,
           });
           ```
         - Add worker event listeners: 'completed' (log job name + duration), 'failed' (log error), 'error' (log error).
         - Return worker reference for graceful shutdown.
       - `stopEventSyncWorker(worker)`: calls worker.close() for graceful shutdown.

    4. Create `backend/src/jobs/syncScheduler.ts`:
       - Import eventSyncQueue from './queue'.
       - `registerSyncJobs()`:
         - If eventSyncQueue is null, log warning and return.
         - Add repeatable job 'scheduled-sync' with cron pattern '0 */4 * * *' (every 4 hours) and jobId 'scheduled-event-sync'.
         - Add repeatable job 'check-cancellations' with cron pattern '0 6 * * *' (daily at 6 AM UTC) and jobId 'daily-cancellation-check'.
         - Log registered jobs.
       - `triggerManualSync(regionId?: string)`: Adds a one-off 'region-sync' job with optional regionId data. For manual triggers / testing.
