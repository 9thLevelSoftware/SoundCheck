# Roadmap: v2.0 Beta Launch

## Overview

v2.0 transforms SoundCheck from a feature-complete development build into a production-ready beta. The Board of Directors (5-0 APPROVE WITH CONDITIONS) identified 28 items across security, UX, data integrity, operations, and social bootstrapping. The critical path is: infrastructure security (unblocks everything) -> mobile UX fixes (first-time experience) -> backend integrity (data correctness) -> operational readiness (monitoring) -> social bootstrapping (beta retention). All phases address existing code — no new architecture or greenfield features.

## Phases

- [x] **Phase 13: Security & Infrastructure Hardening** - Rotate secrets, configure production env, add trust proxy, fix CI, harden auth
- [x] **Phase 14: Mobile UX & Accessibility** - Wire onboarding router, fix legal links, ship dark-only, actionable empty states, a11y
- [ ] **Phase 15: Backend Data Integrity & Logging** - Block filter on Wrapped, VenueService fix, idempotency guard, structured logging
- [ ] **Phase 16: Operational Readiness** - Sentry, uptime monitoring, deployment runbook, smoke tests, staging environment
- [ ] **Phase 17: Social Graph & Beta Onramp** - Friend discovery, share CTA elevation, seed content, RevenueCat API update

## Phase Details

### Phase 13: Security & Infrastructure Hardening
**Goal**: Eliminate all critical security vulnerabilities and configure the production environment so that core features (event sync, push notifications, photo uploads, real-time updates) actually function.
**Depends on**: Nothing (first phase — unblocks everything)
**Requirements**: BETA-01, BETA-02, BETA-03, BETA-04, BETA-05, BETA-06, BETA-07, BETA-08, BETA-09
**Success Criteria** (what must be TRUE):
  1. All secrets rotated and stored exclusively in Railway environment variables — .env file contains no production values
  2. NODE_ENV=production verified — error responses hide stack traces, CORS rejects wildcards, query logging disabled
  3. All third-party integrations functional: event sync (Ticketmaster), push notifications (Firebase), photo uploads (R2), real-time (WebSocket+Redis), subscriptions (RevenueCat webhook), error tracking (Sentry)
  4. Rate limiting uses client IP (not proxy IP) via trust proxy configuration
  5. CI pipeline triggers on master branch pushes and PRs
**Plans**: 2 plans
Plans:
- [x] 13-01-PLAN.md — Trust proxy + fail-closed rate limiting + social auth sentinel replacement + .bak cleanup
- [x] 13-02-PLAN.md — CI pipeline fix + gitleaks secret detection + Railway env var documentation

### Phase 14: Mobile UX & Accessibility
**Goal**: Fix the first-time user experience so beta testers land on a guided, functional app instead of empty screens with no direction.
**Depends on**: Nothing (can run in parallel with Phase 13)
**Requirements**: BETA-10, BETA-11, BETA-12, BETA-13, BETA-14, BETA-15
**Success Criteria** (what must be TRUE):
  1. New users see onboarding carousel + genre picker before login (hasSeenOnboarding check in router redirect)
  2. Terms of Service and Privacy Policy links on subscription screen navigate to real URLs (App Store compliant)
  3. Only dark theme available in settings — no broken light mode visible to beta users
  4. Feed empty states have actionable CTAs (buttons that navigate to discover/search)
  5. Bottom nav bar and check-in button have Semantics labels for screen readers
**Plans**: 2 plans
Plans:
- [x] 14-01-PLAN.md — Onboarding router wiring + legal links + dark-only theme (BETA-10, BETA-11, BETA-12)
- [x] 14-02-PLAN.md — Feed empty states + accessibility + button contrast (BETA-13, BETA-14, BETA-15)

### Phase 15: Backend Data Integrity & Logging
**Goal**: Fix data correctness bugs that could surface during beta testing and replace unstructured console logging with the existing winston logger.
**Depends on**: Phase 13 (needs production env configured for logging to work correctly)
**Requirements**: BETA-16, BETA-17, BETA-18, BETA-19
**Success Criteria** (what must be TRUE):
  1. Wrapped friend overlap stats exclude blocked users (getBlockFilterSQL applied)
  2. VenueService API response field accurately represents the underlying data (reviews vs. checkins resolved)
  3. Owner review responses return 409 if response already exists (no silent overwrites)
  4. Zero console.log/error/warn calls in backend source — all logging through winston utility
**Plans**: 2 plans

### Phase 16: Operational Readiness
**Goal**: Establish the minimum monitoring, alerting, and deployment infrastructure needed to operate a beta with real users. Know when things break before users report them.
**Depends on**: Phase 13 (needs Sentry DSN and Railway env configured)
**Requirements**: BETA-20, BETA-21, BETA-22, BETA-23, BETA-24
**Success Criteria** (what must be TRUE):
  1. Sentry captures backend errors with proper data scrubbing — verified with a test error
  2. Uptime monitoring pings /health endpoint and alerts on downtime within 5 minutes
  3. Deployment runbook documented: deploy, verify, rollback steps
  4. Smoke tests can verify core API paths (auth, check-in, feed) post-deployment
  5. Staging environment exists on Railway with separate database for pre-deploy validation
**Plans**: 2 plans

### Phase 17: Social Graph & Beta Onramp
**Goal**: Give beta users a way to find each other, optimize the viral loop, and ensure the app feels alive on day one with seeded content.
**Depends on**: Phase 14 (mobile UX fixes in place), Phase 15 (backend fixes in place)
**Requirements**: BETA-25, BETA-26, BETA-27, BETA-28
**Success Criteria** (what must be TRUE):
  1. Users can discover and follow other users via search or suggestions
  2. Post-check-in celebration screen prominently features share action as primary CTA
  3. Global feed shows activity from seeded demo accounts on beta launch day
  4. RevenueCat SDK uses non-deprecated purchase API
**Plans**: 2-3 plans
