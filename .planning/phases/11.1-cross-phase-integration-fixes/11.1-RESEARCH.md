# Phase 11.1: Cross-Phase Integration Fixes - Research

**Researched:** 2026-02-28
**Domain:** Backend service integration, PostgreSQL triggers, SQL query patching
**Confidence:** HIGH

## Summary

Phase 11.1 addresses 4 cross-phase integration gaps discovered by the v1.1 milestone audit. All 4 gaps are backend-only fixes -- no new mobile UI or features are required. The mobile code already checks `claimedByUserId` correctly; it just never receives a non-null value from the API. Similarly, feed UIs already display `toast_count`/`comment_count` but these columns are always 0 because no write path maintains them.

The fixes are surgical and well-scoped: (1) a PostgreSQL migration adding triggers to maintain denormalized `toast_count`/`comment_count` on the `checkins` table, (2) adding `claimed_by_user_id` to SELECT lists and mapper functions in VenueService and BandService, (3) importing BlockService into TrendingService and applying `getBlockFilterSQL()` to the LATERAL subqueries, and (4) adding `AND c.is_hidden IS NOT TRUE` to the SearchService event `checkin_count` subquery at line 160.

**Primary recommendation:** Implement as 2 plans -- Plan 01 covers the database migration (triggers + backfill for denormalized counts) and Plan 02 covers the 3 service-level query patches (claimed_by_user_id, block filtering, is_hidden filtering). This ordering ensures the trigger migration runs first so the count columns are correct when feeds are tested.

### Pattern 1: PostgreSQL Trigger for Denormalized Count Maintenance
**What:** Use `CREATE FUNCTION` + `CREATE TRIGGER` to auto-maintain `checkins.toast_count` and `checkins.comment_count` whenever rows are inserted/deleted from `toasts` or `checkin_comments`.
**When to use:** When denormalized aggregate columns must stay in sync with their source tables without application-level coordination.
**Why triggers (not application-level UPDATEs):** Triggers are atomic with the mutation, handle all code paths (including future ones), and cannot be forgotten. The alternative -- adding UPDATE statements to every `toastCheckin()`, `untoastCheckin()`, `addComment()`, `deleteComment()` call -- is fragile and has already been missed once.

```sql
-- Source: PostgreSQL trigger pattern for denormalized counts
-- Migration 037

-- Toast count trigger function
CREATE OR REPLACE FUNCTION update_checkin_toast_count()
RETURNS TRIGGER AS $$
BEGIN
  IF TG_OP = 'INSERT' THEN
    UPDATE checkins SET toast_count = toast_count + 1 WHERE id = NEW.checkin_id;
    RETURN NEW;
  ELSIF TG_OP = 'DELETE' THEN
    UPDATE checkins SET toast_count = GREATEST(toast_count - 1, 0) WHERE id = OLD.checkin_id;
    RETURN OLD;
  END IF;
  RETURN NULL;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_toast_count_insert
  AFTER INSERT ON toasts
  FOR EACH ROW EXECUTE FUNCTION update_checkin_toast_count();

CREATE TRIGGER trg_toast_count_delete
  AFTER DELETE ON toasts
  FOR EACH ROW EXECUTE FUNCTION update_checkin_toast_count();

-- Comment count trigger function
CREATE OR REPLACE FUNCTION update_checkin_comment_count()
RETURNS TRIGGER AS $$
BEGIN
  IF TG_OP = 'INSERT' THEN
    UPDATE checkins SET comment_count = comment_count + 1 WHERE id = NEW.checkin_id;
    RETURN NEW;
  ELSIF TG_OP = 'DELETE' THEN
    UPDATE checkins SET comment_count = GREATEST(comment_count - 1, 0) WHERE id = OLD.checkin_id;
    RETURN OLD;
  END IF;
  RETURN NULL;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_comment_count_insert
  AFTER INSERT ON checkin_comments
  FOR EACH ROW EXECUTE FUNCTION update_checkin_comment_count();

CREATE TRIGGER trg_comment_count_delete
  AFTER DELETE ON checkin_comments
  FOR EACH ROW EXECUTE FUNCTION update_checkin_comment_count();

-- Backfill existing data
UPDATE checkins SET
  toast_count = (SELECT COUNT(*) FROM toasts WHERE checkin_id = checkins.id),
  comment_count = (SELECT COUNT(*) FROM checkin_comments WHERE checkin_id = checkins.id);
```

### Anti-Patterns to Avoid
- **Application-level count updates instead of triggers:** Already failed once (Phase 11 Plan 02 added denormalized columns but no write path). Triggers are the correct solution.
- **Adding block filtering only to the outer WHERE:** The block filter must go inside the LATERAL subqueries that compute velocity and friend signals, not just the outer event filter. Blocked users' check-ins should not contribute to velocity_stats or friend_stats signal computation.
- **Forgetting to include in ALL relevant SELECT lists:** VenueService has multiple SELECT lists (getVenueById, searchVenues, getPopularVenues, getVenuesNear, updateVenue, createVenue). For VERIFY-04 scope, the critical one is `getVenueById()` since that's what the mobile detail screen calls. However, all methods returning `Venue` objects via `mapDbVenueToVenue()` will benefit from the mapper fix since the mapper handles the field regardless of whether the SELECT included it (it maps to `undefined` when absent). To be thorough, add it to all SELECT lists that feed through `mapDbVenueToVenue()`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Denormalized count maintenance | Application-level UPDATE calls in every mutation | PostgreSQL triggers | Atomic, covers all code paths, already missed once |
| Block filtering SQL | Custom WHERE clause per service | `BlockService.getBlockFilterSQL()` | Centralized, UUID-validated, bilateral, already tested |
| is_hidden filtering | Custom logic per query | `IS NOT TRUE` pattern from Phase 9.1 | Null-safe, consistent with all other services |

**Key insight:** All four fixes follow patterns that already exist in the codebase. No novel infrastructure or design is needed.

## Common Pitfalls

### Pitfall 1: Trigger Not Handling GREATEST(0) for Decrements
**What goes wrong:** DELETE trigger decrements count below 0 if data was inconsistent.
**Why it happens:** Race conditions or previous bugs could leave counts at 0 when deleting a toast/comment that actually exists.
**How to avoid:** Use `GREATEST(toast_count - 1, 0)` to floor at 0. The backfill step corrects any initial inconsistency.
**Warning signs:** Negative counts in feed displays.

### Pitfall 2: Missing Backfill in Migration
**What goes wrong:** Triggers only fire on future mutations. Existing check-ins keep toast_count=0 and comment_count=0.
**Why it happens:** Developer adds triggers but forgets the backfill UPDATE for existing data.
**How to avoid:** The migration MUST include a backfill UPDATE after creating triggers. Run the correlated subquery: `UPDATE checkins SET toast_count = (SELECT COUNT(*) FROM toasts WHERE checkin_id = checkins.id), comment_count = (SELECT COUNT(*) FROM checkin_comments WHERE checkin_id = checkins.id)`.
**Warning signs:** Old check-ins still show 0 counts.

### Pitfall 3: Block Filter Placement in TrendingService
**What goes wrong:** Block filter placed in outer WHERE clause but not in LATERAL subqueries.
**Why it happens:** Developer adds filter where events are filtered but not where signals are computed.
**How to avoid:** The `getBlockFilterSQL()` must be added to the `velocity_stats` LATERAL subquery (where `c.user_id` is available) and the `friend_stats` LATERAL subquery (where `er.user_id` is available). The velocity_stats already has `c.user_id` in scope. The friend_stats uses `er.user_id` -- the block filter should exclude RSVPs from blocked users affecting friend signals.
**Warning signs:** Blocked user's check-ins still inflate trending scores even though their events are filtered.

### Pitfall 4: TrendingService Query Parameterization After Block Filter
**What goes wrong:** `getBlockFilterSQL()` injects a raw SQL fragment with the userId. The TrendingService query currently uses only positional parameters `$1-$6`.
**Why it happens:** The block filter pattern interpolates the userId directly (after UUID validation) rather than using a positional parameter.
**How to avoid:** Insert the block filter SQL string via template literal interpolation, exactly as FeedService does. Do NOT try to add it as a positional parameter -- BlockService.getBlockFilterSQL() returns a complete SQL `AND NOT EXISTS(...)` fragment.

### Pitfall 5: Forgetting claimed_by_user_id in RETURNING Clauses
**What goes wrong:** After updating or creating a venue/band, the RETURNING clause doesn't include `claimed_by_user_id`, so the response is missing it.
**Why it happens:** Multiple SELECT/RETURNING lists exist in VenueService and BandService.
**How to avoid:** Add `claimed_by_user_id` to ALL SELECT and RETURNING clauses in VenueService and BandService. The mapper will handle it uniformly.

## Open Questions

1. **Should claimed_by_user_id be added to ALL VenueService/BandService SELECT lists or just getById?**
   - What we know: getVenueById/getBandById are what the detail screens call. searchVenues/searchBands/getPopular/getTrending also use the mapper.
   - What's unclear: Whether list views need the badge too.
   - Recommendation: Add to ALL SELECT lists for consistency. The mapper handles it uniformly, and list views may want to show verification badges in the future. The cost is negligible (one UUID column).

2. **Should the backfill UPDATE be a single statement or batched?**
   - What we know: The backfill uses a correlated subquery that updates all rows.
   - What's unclear: How many checkins exist in the database.
   - Recommendation: Single UPDATE statement. For a pre-launch app, the data volume is small. If this were a production system with millions of rows, batching would be needed. The migration runner wraps this in a transaction.
