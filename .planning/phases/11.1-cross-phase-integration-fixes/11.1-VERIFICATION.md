### Summary

Phase 11.1 achieves its goal. All five cross-phase integration gaps are closed:

1. **Denormalized count write path (SCALE-02):** Migration 037 installs four PostgreSQL triggers (AFTER INSERT/DELETE on `toasts` and `checkin_comments`) that atomically maintain `toast_count` and `comment_count` on the `checkins` table, with GREATEST-flooring to prevent negative counts. A backfill UPDATE corrects all historical data. The read-write gap that left these columns permanently zero is closed.

2. **Verification badge data (VERIFY-04):** `claimed_by_user_id` is now included in all six VenueService and seven BandService SELECT/RETURNING clauses that flow through their mappers. Both mappers output `claimedByUserId: row.claimed_by_user_id || undefined`, which serializes the field as non-null JSON only for claimed entities — exactly what mobile verification badge display requires.

3. **Block filtering in trending (SAFE-04):** TrendingService imports and instantiates BlockService, then applies `getBlockFilterSQL()` to both signal-producing LATERAL subqueries (velocity_stats on `c.user_id`, friend_stats on `er.user_id`). Blocked users can no longer inflate trending scores for either check-in velocity or friend presence signals.

4. **Hidden content filtering in search (SAFE-02):** SearchService's event `checkin_count` subquery now includes `AND c.is_hidden IS NOT TRUE`, using the null-safe IS NOT TRUE pattern established in Phase 9.1. Moderated check-ins no longer inflate search event counts.
