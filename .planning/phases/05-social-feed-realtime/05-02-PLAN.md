Use the same pattern: getRedisUrl(), new IORedis with maxRetriesPerRequest: null, enableReadyCheck: false. This connection is SEPARATE from the cache connection because once a connection enters subscriber mode it can only run subscribe/unsubscribe.

    Also add a `getPublisherConnection()` function that returns the existing Redis connection from redisRateLimiter.ts (via getRedis()). This is for publishing -- publishing does NOT require a dedicated connection.

    **websocket.ts - Add Redis Pub/Sub subscription:**
    In the WebSocketServer class:
    1. Add a private `subscriber` field of type IORedis | null
    2. In the `init()` method, AFTER the WebSocket server is set up, create the Pub/Sub subscriber:
       ```
       try {
         this.subscriber = createPubSubConnection();
         this.subscriber.subscribe('checkin:new');
         this.subscriber.on('message', (channel, message) => {
           if (channel === 'checkin:new') {
             this.handleCheckinPubSub(JSON.parse(message));
           }
         });
         console.log('Redis Pub/Sub subscriber connected for WebSocket fan-out');
       } catch (err) {
         console.warn('Redis Pub/Sub not available, WebSocket fan-out disabled');
         this.subscriber = null;
       }
       ```
    3. Add `handleCheckinPubSub(data)` private method:
       - data shape: { type: 'new_checkin', checkin: FeedItem-like object, followerIds: string[], eventId: string, creatorEventIds: string[] }
       - For each followerId in data.followerIds, call `this.sendToUser(followerId, 'new_checkin', data.checkin)`
       - **Same-event detection:** For each followerId, check if they are in a room matching `event:${data.eventId}` (using getRoomUsers). If so, send a DIFFERENT event: `this.sendToUser(followerId, 'same_event_checkin', { ...data.checkin, message: '${data.checkin.username} is here too!' })`
       - Log: `Fan-out new_checkin to ${data.followerIds.length} followers`
    4. In the `close()` method, also close the subscriber connection: `this.subscriber?.quit()`
    5. Add `SAME_EVENT_CHECKIN: 'same_event_checkin'` to WebSocketEvents object

    **notificationQueue.ts:**
    Follow the exact pattern from badgeQueue.ts. Create BullMQ Queue named 'notification-batch'. Graceful degradation: null if REDIS_URL not set. Default job options: 3 attempts, exponential backoff 5000ms, removeOnComplete 500, removeOnFail 200.

    **notificationWorker.ts:**
    Follow the exact pattern from badgeWorker.ts. Create Worker for 'notification-batch' queue.

    Worker processor:
    1. Extract userId from job.data
    2. Get Redis instance via getRedis()
    3. LRANGE `notif:batch:${userId}` 0 -1 to get all pending notification items
    4. DEL `notif:batch:${userId}` to clear the list
    5. If empty list, return (no-op gracefully -- this handles the race condition)
    6. Parse each item as JSON (shape: { username, eventName, venueName })
    7. If 1 item: send direct notification with FOMO tone: title="${username} checked in!", body="At ${eventName} @ ${venueName}"
    8. If multiple items: send summary: title="${count} friends checked in!", body="${first.username} and ${count-1} others are at shows tonight"
    9. Use PushNotificationService.sendToUser() for FCM delivery
    10. Concurrency: 5 (notification sending is I/O-bound, not rate-limited)

    Export startNotificationWorker() and stopNotificationWorker() functions.

    **CheckinService.ts - Enqueue notification batch:**
    In createEventCheckin(), AFTER the Pub/Sub publish (from Task 1):
    1. Import notificationQueue from notificationQueue.ts
    2. For each followerId, enqueue a notification batch:
       ```
       const redis = getRedis();
       const listKey = `notif:batch:${followerId}`;
       await redis?.rpush(listKey, JSON.stringify({
         username: checkin.user?.username || 'Someone',
         eventName: event.eventName || 'a show',
         venueName: venue?.name || '',
       }));
       await redis?.expire(listKey, 300); // 5min safety TTL
       await notificationQueue?.add('send-batch', { userId: followerId }, {
         delay: 120_000, // 2-minute batching window
         jobId: `notif-batch:${followerId}`, // dedup: one job per user per window
       });
       ```

For now, just add a TODO comment noting that badge-earned push notifications should be triggered from the badge eval worker completion.
    4. Fire-and-forget: wrap in try/catch, don't block check-in response
