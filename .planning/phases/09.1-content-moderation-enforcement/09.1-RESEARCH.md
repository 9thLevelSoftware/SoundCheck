## Summary

Phase 9 built the full moderation pipeline: users report content, photos are scanned by Cloud Vision SafeSearch, flagged content is auto-hidden (`is_hidden = true`), and admins can approve or remove content from the moderation queue. The `is_hidden` column exists on both `checkins` and `checkin_comments` tables (migration 030). **However, no content-serving query reads this column.** Hidden content remains visible in every feed.

This is a pure backend SQL fix -- every query that returns checkins or comments to end-users needs a `WHERE` clause filtering out `is_hidden = true` rows. No new libraries, no new tables, no new endpoints. The pattern is identical to how `BlockService.getBlockFilterSQL()` was wired into feed queries during Phase 9 Plan 03.

**Primary recommendation:** Add `AND (c.is_hidden IS NOT TRUE)` to every checkin query and `AND (cc.is_hidden IS NOT TRUE)` to every comment query in user-facing services, add a partial index for performance, and invalidate relevant feed caches when content is hidden/unhidden.

## Architecture Patterns

### Pattern 1: SQL Filter Injection (Established Pattern)

**What:** Append a `WHERE` clause fragment to existing queries to exclude filtered content.
**When to use:** Whenever content must be conditionally excluded from user-facing queries.
**Already used by:** `BlockService.getBlockFilterSQL()` -- returns an `AND NOT EXISTS(...)` fragment appended to every content query.

**Example (existing block filter pattern from FeedService line 120):**
```typescript
// BlockService returns: AND NOT EXISTS (SELECT 1 FROM user_blocks WHERE ...)
WHERE uf.follower_id = $1
  ${this.blockService.getBlockFilterSQL(userId, 'c.user_id')}
```

**For is_hidden, the equivalent is simpler -- a direct column check:**
```sql
-- For checkins (alias c):
AND (c.is_hidden IS NOT TRUE)

-- For checkin_comments (alias c or cm or cc):
AND (cm.is_hidden IS NOT TRUE)
```

**Why `IS NOT TRUE` instead of `= FALSE`:** Handles NULL values safely. If `is_hidden` is NULL (shouldn't happen with DEFAULT FALSE, but defensive), `IS NOT TRUE` still returns the row. `= FALSE` would exclude NULLs.

### Pattern 2: Partial Index for Performance

**What:** A partial PostgreSQL index that covers only `is_hidden = true` rows.
**When to use:** When the filtered column has highly skewed distribution (very few TRUE, many FALSE).
**Why:** Full index on `is_hidden` wastes space since 99%+ of rows are FALSE. A partial index is tiny and speeds up the planner's ability to confirm the filter is cheap.

```sql
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_checkins_hidden
  ON checkins (id) WHERE is_hidden = true;

CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_comments_hidden
  ON checkin_comments (id) WHERE is_hidden = true;
```

**Note:** These indexes help the planner optimize `AND (c.is_hidden IS NOT TRUE)` in large tables. At current scale they're optional but cheap to add proactively.

### Pattern 3: Cache Invalidation on Moderation Action

**What:** When `ModerationService.autoHideContent()` or `reviewItem()` (with 'approved' action) changes `is_hidden`, the relevant feed caches must be invalidated so users see the change on next refresh.
**When to use:** After any write to `is_hidden`.
**How:** Call `FeedService.invalidateUserFeedCache()` for the content author's followers and `FeedService.invalidateEventFeedCache()` for the event. This is the same pattern used by `CheckinCreatorService` after creating/deleting a check-in.

### Anti-Patterns to Avoid

- **Filtering only in the application layer:** Do NOT fetch all rows and filter in TypeScript. The `WHERE` clause must be in SQL so the database can use the index and avoid transmitting hidden rows.
- **Creating a reusable SQL fragment like BlockService:** Overkill for a simple boolean column. `AND (c.is_hidden IS NOT TRUE)` is one line, always the same, and doesn't need parameterization. A helper function would add indirection without benefit.
- **Forgetting comment_count subqueries:** Several queries count comments via `COUNT(DISTINCT cm.id)`. If `cm` is joined without filtering, the count includes hidden comments. The `LEFT JOIN` must also filter: `LEFT JOIN checkin_comments cm ON c.id = cm.checkin_id AND cm.is_hidden IS NOT TRUE`.
- **Forgetting toast_count with hidden checkins:** Badge evaluators and stats queries count checkins. If a hidden checkin still counts toward badges or stats, the user retains credit for moderated content. This may be intentional (badges earned are not revoked) or unintentional -- a design decision to surface to the planner.

## Common Pitfalls

### Pitfall 1: Missing a Query

**What goes wrong:** One content-serving query is missed, and hidden content leaks through that specific endpoint.
**Why it happens:** Queries are spread across 6+ service files, some with multiple methods.
**How to avoid:** Use the exhaustive inventory below. Every query that touches `checkins` or `checkin_comments` and returns data to users must be checked.
**Warning signs:** grep for `FROM checkins` and `FROM checkin_comments` across all service files; any query without `is_hidden` filter is a leak.

### Pitfall 2: Counting Hidden Content in Aggregates

**What goes wrong:** `comment_count` or `toast_count` includes hidden items; user sees "3 comments" but only 2 are visible.
**Why it happens:** The `LEFT JOIN checkin_comments cm ON c.id = cm.checkin_id` in feed queries joins ALL comments. `COUNT(DISTINCT cm.id)` then counts hidden ones.
**How to avoid:** Add filter to the JOIN condition: `LEFT JOIN checkin_comments cm ON c.id = cm.checkin_id AND cm.is_hidden IS NOT TRUE`.

### Pitfall 3: Cache Serving Stale Hidden Content

**What goes wrong:** Content is hidden but the cached feed page still contains it. Users see hidden content for up to 60 seconds (friends feed TTL).
**Why it happens:** `ModerationService.autoHideContent()` writes to DB but does not invalidate feed caches.
**How to avoid:** After writing `is_hidden = true`, look up the content author and event, then call cache invalidation. Same pattern as `CheckinCreatorService`.

### Pitfall 4: getCheckinById Returning Hidden Content

**What goes wrong:** Deep-linked or bookmarked hidden check-in is still directly accessible via its ID.
**Why it happens:** `getCheckinById()` fetches by primary key with no visibility check.
**How to avoid:** Add `AND (c.is_hidden IS NOT TRUE)` to the WHERE clause, OR let it return the checkin but mark it as hidden for the caller. Design decision: return 404 (hides existence) or return with `isHidden: true` flag (lets owner know why it's gone from feeds).

### Pitfall 5: Badge/Stats Counting Hidden Check-ins

**What goes wrong:** User earns a "10 Shows" badge partly from check-ins that were later hidden for policy violations.
**Why it happens:** `BadgeEvaluators` count all checkins for a user regardless of visibility.
**How to avoid:** Decision point -- should hidden check-ins count toward badges and stats? Arguments both ways:
- **Yes (count them):** User attended the show; hiding the photo doesn't erase the check-in. Revoking badges creates confusion.
- **No (exclude them):** If content is hidden for policy violation, the whole check-in may be fraudulent.
**Recommendation:** Count hidden check-ins for badges/stats (don't revoke earned rewards) but exclude them from user-facing feeds. This matches the common pattern in social apps (Instagram doesn't revoke follower counts when posts are removed).

### Example 3: ModerationService -- Add Cache Invalidation

```typescript
// In autoHideContent(), after the UPDATE query:
async autoHideContent(contentType: ContentType, contentId: string): Promise<void> {
  // ... existing switch/UPDATE logic ...

  // Invalidate feed caches so hidden content disappears on next refresh
  if (contentType === 'checkin' || contentType === 'photo') {
    const result = await this.db.query(
      'SELECT user_id, event_id FROM checkins WHERE id = $1',
      [contentId]
    );
    if (result.rows[0]) {
      const feedService = new FeedService();
      // Get all followers of the content author
      const followers = await this.db.query(
        'SELECT follower_id FROM user_followers WHERE following_id = $1',
        [result.rows[0].user_id]
      );
      for (const f of followers.rows) {
        await feedService.invalidateUserFeedCache(f.follower_id);
      }
      if (result.rows[0].event_id) {
        await feedService.invalidateEventFeedCache(result.rows[0].event_id);
      }
    }
  }
}
```

## Open Questions

1. **Should `getCheckinById()` return 404 for hidden content?**
   - What we know: Currently returns the checkin regardless of visibility. Deep links and API calls can access hidden content.
   - What's unclear: Should the content owner still see their own hidden content (with a "hidden" indicator)?
   - Recommendation: Return 404 for all users except the content owner. Owner sees it with an `isHidden: true` flag. This is the Instagram/Twitter pattern. However, for simplicity in Phase 9.1, returning 404 universally is acceptable -- the owner learns through moderation notification (future phase).
   - **Planner decision:** Start with simple 404 for all users. Owner visibility can be added later if needed.

2. **Should hidden check-ins count toward badges and stats?**
   - What we know: Badge evaluators and StatsService count all checkins without filtering.
   - What's unclear: Is hiding a photo (the common moderation case) sufficient, or should the entire check-in not count?
   - Recommendation: **Do NOT filter `is_hidden` in badge evaluators or StatsService.** Rationale: the check-in itself (attending a show) is not fraudulent when only the photo is flagged. Revoking badges creates user confusion. Stats should reflect actual attendance.
   - **Planner decision:** Leave badges/stats unfiltered. Only filter user-facing content feeds and discovery.

3. **Cache invalidation scope for `autoHideContent()`**
   - What we know: Feed caches have 30-60s TTL. Without explicit invalidation, hidden content lingers briefly.
   - What's unclear: Is TTL-based expiry (30-60s) acceptable, or must invalidation be immediate?
   - Recommendation: Add explicit cache invalidation in `ModerationService` for immediate effect. The code pattern already exists (`FeedService.invalidateUserFeedCache`). Cost is low (a few Redis DEL calls).
   - **Planner decision:** Add cache invalidation for completeness. Success criterion #4 says "immediate removal on next refresh."
