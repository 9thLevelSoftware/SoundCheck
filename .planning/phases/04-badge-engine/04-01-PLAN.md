**1. Create `backend/src/services/BadgeEvaluators.ts`** -- the evaluator registry.

Define the interface and registry:
```typescript
interface EvalResult {
  current: number;
  target: number;
  earned: boolean;
  metadata?: Record<string, any>; // e.g., superfan band info
}
type BadgeEvaluator = (userId: string, criteria: Record<string, any>) => Promise<EvalResult>;
const evaluatorRegistry: Map<string, BadgeEvaluator> = new Map();
```

Register 6 evaluators:

- `checkin_count`: `SELECT COUNT(*) FROM checkins WHERE user_id = $1`. Compare against `criteria.threshold`.

- `genre_explorer`: Join checkins -> events -> event_lineup -> bands. `SELECT COUNT(DISTINCT c.event_id) FROM checkins c JOIN event_lineup el ON c.event_id = el.event_id JOIN bands b ON el.band_id = b.id WHERE c.user_id = $1 AND LOWER(b.genre) = LOWER($2)`. Use `criteria.genre` and `criteria.threshold`.

- `unique_venues`: `SELECT COUNT(DISTINCT venue_id) FROM checkins WHERE user_id = $1`. Compare against `criteria.threshold`.

- `superfan`: Find the band the user has seen most times. `SELECT el.band_id, b.name as band_name, COUNT(DISTINCT c.event_id) as times_seen FROM checkins c JOIN event_lineup el ON c.event_id = el.event_id JOIN bands b ON el.band_id = b.id WHERE c.user_id = $1 GROUP BY el.band_id, b.name ORDER BY times_seen DESC LIMIT 1`. Return `current = times_seen` of top band, `target = criteria.threshold`. Include `{bandId, bandName}` in metadata if earned.

- `festival_warrior`: Max check-ins on any single day. `SELECT DATE(created_at) as d, COUNT(*) as cnt FROM checkins WHERE user_id = $1 GROUP BY DATE(created_at) ORDER BY cnt DESC LIMIT 1`. Compare max cnt against `criteria.threshold`.

- `road_warrior`: `criteria.field` is either `'city'` or `'state'`. Use parameterized column selection (NOT string interpolation -- use a safe mapping: `const col = criteria.field === 'state' ? 'v.state' : 'v.city'`). `SELECT COUNT(DISTINCT ${col}) FROM checkins c JOIN venues v ON c.venue_id = v.id WHERE c.user_id = $1 AND ${col} IS NOT NULL`. Compare against `criteria.threshold`.

Use `Database.getInstance()` for all queries. Export `evaluatorRegistry`, `EvalResult`, `BadgeEvaluator`.

Must add a `metadata` column check: if `user_badges` does not have a `metadata` column, the INSERT should omit it. If user_badges lacks metadata column, this will fail -- so add column in a migration within plan 02. For now, keep metadata as a parameter but do NOT include it in SQL. Just INSERT user_id and badge_id as before.

- **NEW** `evaluateAndAward(userId: string): Promise<Badge[]>` -- the main evaluation method:
  1. Load all badge definitions: `SELECT id, name, badge_type, requirement_value, criteria, description, icon_url, color FROM badges WHERE criteria IS NOT NULL AND criteria != '{}'::jsonb`
  2. Load user's existing badge IDs: `SELECT badge_id FROM user_badges WHERE user_id = $1`
  3. For each unearned badge definition: look up evaluator by `criteria.type` from evaluatorRegistry. If evaluator exists, call it. If `earned === true`, award badge and collect it.
  4. For each newly awarded badge: create notification via NotificationService (`type: 'badge_earned'`, include badgeId) and send WebSocket event via `sendToUser(userId, 'badge_earned', { badgeId, badgeName, badgeColor, badgeIconUrl })`.
  5. Return array of newly earned badges.

  **IMPORTANT for N+1 optimization:** Group badge definitions by `criteria.type`. For each type, run the evaluator ONCE (it returns a single {current, target, earned}). Then match against all badges of that type. Exception: `genre_explorer` needs one query per genre -- group genre badges by genre value, run one query per genre. This keeps total queries to roughly: 1 (checkin_count) + N_genres (genre_explorer) + 1 (unique_venues) + 1 (superfan) + 1 (festival_warrior) + 1 (road_warrior) = ~10-12 queries, not 30+.

  1. Load all badge definitions with criteria
  2. Load user's earned badge IDs
  3. Group badges by criteria.type
  4. For each type group, run evaluator once to get current count
  5. Return array of `{ badge, currentValue: result.current, requirementValue: result.target, isEarned }` matching the existing BadgeProgress response shape. For genre_explorer, run per-genre.

**1. Create `backend/src/jobs/badgeQueue.ts`** -- clone the pattern from `queue.ts`.

```typescript
import { Queue } from 'bullmq';
import { createBullMQConnection, getRedisUrl } from '../config/redis';

let badgeEvalQueue: Queue | null = null;

try {
  getRedisUrl();
  badgeEvalQueue = new Queue('badge-eval', {
    connection: createBullMQConnection(),
    defaultJobOptions: {
      attempts: 3,
      backoff: { type: 'exponential', delay: 5000 },
      removeOnComplete: { count: 500 },
      removeOnFail: { count: 200 },
    },
  });
  console.log('[BadgeEvalQueue] Badge evaluation queue initialized');
} catch {
  console.warn('[BadgeEvalQueue] REDIS_URL not configured. Badge evaluation queue is disabled.');
  badgeEvalQueue = null;
}

export { badgeEvalQueue };
```

**3. Modify `backend/src/services/CheckinService.ts`** -- add badge job enqueue.

At the top, add import: `import { badgeEvalQueue } from '../jobs/badgeQueue';`

In `createEventCheckin()`, AFTER the successful check-in INSERT and AFTER the promoteIfVerified call (around line 240, just before the return statement), add:

```typescript
// Enqueue async badge evaluation (30-second delay for anti-farming)
if (badgeEvalQueue) {
  try {
    await badgeEvalQueue.add(
      'evaluate',
      { userId, checkinId },
      {
        delay: 30000,
        jobId: `badge-eval-${userId}-${checkinId}`,
      }
    );
  } catch (err) {
    console.error('Warning: failed to enqueue badge evaluation:', err);
    // Non-fatal -- check-in succeeds even if badge queue fails
  }
}
```

This is NON-BLOCKING and NON-FATAL. If Redis is down or queue fails, the check-in still succeeds.
