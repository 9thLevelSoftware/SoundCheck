# Phase 10.2: Password Reset Mobile Fix - Research

**Researched:** 2026-02-27
**Domain:** Flutter mobile deep linking + API URL bug fix
**Confidence:** HIGH

## Summary

This phase closes a gap-closure audit finding: the password reset flow is broken on mobile due to two distinct bugs, plus the deep link handler needed to complete the reset flow from email does not exist yet.

The backend (PasswordResetService, EmailService, PasswordResetController, passwordResetRoutes) is fully implemented and correct. The email sends a `soundcheck://reset-password?token=X` deep link. The mobile side has three problems: (1) `forgot_password_screen.dart` sends requests to `/api/auth/forgot-password` but `DioClient.baseUrl` already includes `/api`, causing a double-prefix 404; (2) `discover_providers.dart` has the same double-prefix bug at `/api/search/users`; (3) there is no route in `app_router.dart` for `/reset-password`, no deep link configuration in AndroidManifest.xml or Info.plist, and no `ResetPasswordScreen` widget exists.

**Primary recommendation:** Fix the two double-prefix URL bugs (simple string changes), add platform deep link configuration for the `soundcheck://` scheme, add a `/reset-password` route to `app_router.dart` with query parameter extraction, and create a new `ResetPasswordScreen` widget that accepts a token and posts to the backend's reset-password endpoint.

### Pattern 1: DioClient URL Convention
**What:** All API paths passed to DioClient are relative to the baseUrl (`https://host/api`). They must NOT include the `/api` prefix.
**When to use:** Every API call in the mobile app.
**Correct pattern:**
```dart
// baseUrl = 'https://soundcheck-app.up.railway.app/api'
// Backend route: app.use('/api/auth', passwordResetRoutes);
// Route handler: router.post('/forgot-password', ...)
// Full backend URL: /api/auth/forgot-password
// DioClient path (strip /api prefix): '/auth/forgot-password'

final response = await dioClient.post(
  '/auth/forgot-password',  // NOT '/api/auth/forgot-password'
  data: {'email': email},
);
```
**Evidence:** `auth_repository.dart` uses `'${ApiConfig.auth}/login'` which resolves to `'/users/login'` -- no `/api` prefix. Every other repository follows this pattern.

### Pattern 2: GoRouter Route with Query Parameters for Deep Links
**What:** GoRouter routes can extract query parameters from deep link URLs via `state.uri.queryParameters`.
**When to use:** When a deep link URL includes query parameters (e.g., `soundcheck://reset-password?token=X`).
**Example:**
```dart
GoRoute(
  path: '/reset-password',
  name: 'reset-password',
  pageBuilder: (context, state) {
    final token = state.uri.queryParameters['token'] ?? '';
    return CustomTransitionPage(
      key: state.pageKey,
      child: ResetPasswordScreen(token: token),
      // ... transitions
    );
  },
),
```

### Pattern 3: Auth Guard Exception for Reset Password
**What:** The router redirect must allow unauthenticated access to `/reset-password` (same as `/forgot-password`, `/login`, `/register`).
**When to use:** Deep link arrives when user is not logged in.
**Example:**
```dart
final isOnAuthPage = state.matchedLocation.startsWith('/login') ||
    state.matchedLocation.startsWith('/register') ||
    state.matchedLocation.startsWith('/forgot-password') ||
    state.matchedLocation.startsWith('/reset-password');  // ADD THIS
```

### Pattern 4: ConsumerStatefulWidget Screen Pattern
**What:** All form-based auth screens (LoginScreen, RegisterScreen, ForgotPasswordScreen) use ConsumerStatefulWidget with local `_isLoading`, `_formKey`, error handling via SnackBar.
**When to use:** ResetPasswordScreen should follow identical pattern.
**Evidence:** `forgot_password_screen.dart` is the closest structural reference.

### Anti-Patterns to Avoid
- **Double `/api` prefix:** Never include `/api` in paths passed to DioClient -- baseUrl already includes it. This is the root cause of the current 404 bug.
- **Using app_links package when go_router suffices:** go_router already handles deep links via Flutter's built-in Router API. Adding app_links creates conflicts requiring `FlutterDeepLinkingEnabled: false` workarounds.
- **Hardcoded URLs in screens:** Use DioClient (not raw Dio) and follow the repository pattern. ForgotPasswordScreen currently calls DioClient directly (acceptable for a simple one-off call), but the URL must be correct.

## Common Pitfalls

### Pitfall 1: Double `/api` Prefix
**What goes wrong:** `DioClient.baseUrl` is `https://host/api`. Passing `/api/auth/forgot-password` results in `https://host/api/api/auth/forgot-password` which returns 404.
**Why it happens:** Developer confused backend route mounting path with DioClient path convention.
**How to avoid:** All DioClient paths must omit the `/api` prefix. Check `api_config.dart` for baseUrl before writing paths.
**Warning signs:** 404 errors on API calls that should work; the path `/api` appears twice in network logs.

### Pitfall 2: Missing Auth Guard Exception for Deep Link Route
**What goes wrong:** User clicks deep link from email, app opens, GoRouter redirect sends them to `/login` because they're not authenticated and `/reset-password` is not in the auth page exception list.
**Why it happens:** The redirect logic checks `isOnAuthPage` against a hardcoded list of paths; new auth-related routes must be added.
**How to avoid:** Add `/reset-password` to the `isOnAuthPage` check in the redirect callback.
**Warning signs:** Deep link opens app but user sees login screen instead of reset form.

### Pitfall 3: Missing Platform Deep Link Configuration
**What goes wrong:** OS doesn't know to route `soundcheck://` URLs to the app. Clicking the email link shows an error or opens a browser.
**Why it happens:** AndroidManifest.xml has no intent-filter for the custom scheme; Info.plist has no CFBundleURLTypes entry.
**How to avoid:** Add platform configuration for both Android and iOS.
**Warning signs:** Deep link works in debug (adb) but not from actual email client.

### Pitfall 4: Token Not Passed from Deep Link to Screen
**What goes wrong:** ResetPasswordScreen renders but has no token, so the backend rejects the reset request.
**Why it happens:** GoRoute doesn't extract query parameters from the deep link URL.
**How to avoid:** Use `state.uri.queryParameters['token']` in the route builder.
**Warning signs:** Screen shows but submitting new password returns "Invalid or expired reset token".

### Pitfall 5: Missing `flutter_deeplinking_enabled` Metadata
**What goes wrong:** Flutter's deep link handler doesn't process incoming URLs.
**Why it happens:** Android requires `flutter_deeplinking_enabled` meta-data tag set to `true` in AndroidManifest.xml for the Router API to receive deep links.
**How to avoid:** Add the metadata tag. On iOS, add `FlutterDeepLinkingEnabled: true` to Info.plist.
**Warning signs:** Deep link opens app but GoRouter never receives the URL.

### Android Deep Link Configuration (AndroidManifest.xml)
```xml
<!-- Inside <activity> tag, after existing intent-filter -->
<meta-data
    android:name="flutter_deeplinking_enabled"
    android:value="true" />
<intent-filter>
    <action android:name="android.intent.action.VIEW" />
    <category android:name="android.intent.category.DEFAULT" />
    <category android:name="android.intent.category.BROWSABLE" />
    <data android:scheme="soundcheck" />
</intent-filter>
```

## Open Questions

1. **Custom scheme uniqueness**
   - What we know: `soundcheck://` is a custom URL scheme. Any app could register the same scheme and intercept links.
   - What's unclear: Whether another app on the user's device uses `soundcheck://`.
   - Recommendation: Custom scheme is fine for password reset (token is single-use, expires in 1 hour, hashed in DB). For production hardening later, consider Android App Links / iOS Universal Links with domain verification. Out of scope for this phase.

2. **Email client compatibility with custom schemes**
   - What we know: Most email clients (Gmail, Apple Mail, Outlook) open custom scheme links. Some webmail clients may not.
   - What's unclear: Edge cases with specific email clients stripping or blocking custom scheme URLs.
   - Recommendation: The backend's email template uses an `<a href>` tag with the `soundcheck://` URL. This is the standard approach. If issues arise, a future phase could add a web intermediary redirect page (e.g., `https://soundcheck-app.up.railway.app/reset?token=X` that redirects to `soundcheck://reset-password?token=X`). Out of scope for this phase.
