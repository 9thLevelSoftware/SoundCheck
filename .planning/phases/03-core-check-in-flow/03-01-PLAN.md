**CheckinService.ts -- Add new method `createEventCheckin` (keep existing `createCheckin` for backward compat):**

1. New interface `CreateEventCheckinRequest`:
   ```
   {
     userId: string;
     eventId: string;
     locationLat?: number;
     locationLon?: number;
     comment?: string;
     vibeTagIds?: string[];
   }
   ```

2. `createEventCheckin(data)` method:
   - Validate event exists and is not cancelled: `SELECT e.*, v.latitude, v.longitude, v.venue_type, v.timezone FROM events e JOIN venues v ON e.venue_id = v.id WHERE e.id = $1 AND e.is_cancelled = FALSE`
   - If event not found or cancelled, throw Error('Event not found or cancelled')
   - Validate time window using `isWithinTimeWindow(event)` helper (see below)
   - Determine `is_verified` via non-blocking location check using `verifyLocation()` helper
   - INSERT into checkins with: `user_id, event_id, venue_id (from event), is_verified, review_text, checkin_latitude, checkin_longitude`. Set `band_id` to headliner band_id from event_lineup for backward compat. Set `event_date` from event.event_date, `rating` to 0.
   - Catch unique constraint violation (error.code === '23505', error.constraint includes 'user_event') and throw Error('You have already checked in to this event') with a flag to return 409
   - After successful insert, call `this.eventService.promoteIfVerified(eventId)` to check organic verification for user-created events
   - Return full checkin via `getCheckinById`

3. Helper `verifyLocation(userLat, userLon, venueLat, venueLon, venueType)`:
   - Returns boolean. Uses Haversine formula (same as VenueService pattern):
     `6371 * acos(cos(radians(lat1)) * cos(radians(lat2)) * cos(radians(lon2) - radians(lon1)) + sin(radians(lat1)) * sin(radians(lat2)))`
   - Radius by venue type: stadium/arena = 2.0km, outdoor = 1.5km, concert_hall/theater = 0.5km, club/bar = 0.3km, default = 1.0km
   - If venueLat/venueLon is null, return false (unverifiable)
   - If userLat/userLon not provided, return false (user didn't share location)

4. Helper `isWithinTimeWindow(event)`:
   - event has: event_date (DATE), doors_time (TIME nullable), start_time (TIME nullable), end_time (TIME nullable), timezone (from venue join)
   - CRITICAL: Use venue timezone for comparison. If timezone is available:
     - Construct window start: doors_time or (start_time - 2 hours) or '16:00' as default, in venue timezone
     - Construct window end: (end_time + 60 min buffer) or (start_time + 6 hours) or '23:59' in venue timezone
     - Compare against current time in venue timezone
   - If no timezone: compare using UTC with generous window (all day on event_date)
   - If no times at all: allow all-day window (00:00 to 23:59 on event_date)
   - For timezone conversion in Node.js: Use `new Date().toLocaleString('en-US', { timeZone: venue.timezone })` to get venue-local time, then compare
   - Return boolean

5. New method `addRatings(checkinId, userId, ratings)`:
   - `ratings` interface: `{ bandRatings?: { bandId: string; rating: number }[]; venueRating?: number }`
   - Verify checkin belongs to userId: `SELECT user_id, event_id FROM checkins WHERE id = $1`
   - If venueRating provided: `UPDATE checkins SET venue_rating = $1 WHERE id = $2` (validate 0.5-5.0, step 0.5)
   - If bandRatings provided: For each bandRating, validate rating 0.5-5.0 and step 0.5, verify band is in event lineup:
     `SELECT 1 FROM event_lineup WHERE event_id = $1 AND band_id = $2`
     Then `INSERT INTO checkin_band_ratings (checkin_id, band_id, rating) VALUES ($1, $2, $3) ON CONFLICT (checkin_id, band_id) DO UPDATE SET rating = $3`
   - Also set `rating` column on checkins to the average of all band ratings (for backward compat with old queries/triggers)
   - Return updated checkin via `getCheckinById`

6. Keep existing `createCheckin` method intact for backward compatibility. Add a check at the top: if `eventId` is present in request body (update CreateCheckinRequest to include optional eventId), delegate to `createEventCheckin` instead.

7. Update `mapDbCheckinToCheckin` to also populate `bandRatings` array from checkin_band_ratings when loading a single checkin. Add a query in `getCheckinById` to fetch band ratings:
   `SELECT cbr.band_id, cbr.rating, b.name as band_name FROM checkin_band_ratings cbr JOIN bands b ON cbr.band_id = b.id WHERE cbr.checkin_id = $1`

**EventService.ts -- Add `getNearbyEvents` method:**

1. Method signature: `getNearbyEvents(lat: number, lon: number, radiusKm: number = 10, limit: number = 20)`
2. SQL query following existing VenueService.getVenuesNear() Haversine pattern:
   ```sql
   SELECT e.*, v.name as venue_name, v.city as venue_city, v.state as venue_state,
          v.latitude as venue_lat, v.longitude as venue_lon, v.venue_type,
          v.image_url as venue_image,
          (6371 * acos(cos(radians($1)) * cos(radians(v.latitude)) *
           cos(radians(v.longitude) - radians($2)) +
           sin(radians($1)) * sin(radians(v.latitude)))) AS distance_km
   FROM events e
   JOIN venues v ON e.venue_id = v.id
   WHERE e.event_date = CURRENT_DATE
     AND e.is_cancelled = FALSE
     AND v.latitude IS NOT NULL AND v.longitude IS NOT NULL
   HAVING (6371 * acos(...)) <= $3
   ORDER BY distance_km ASC
   LIMIT $4
   ```
   NOTE: PostgreSQL doesn't support HAVING without GROUP BY on a non-aggregate. Use a subquery or WHERE with the distance formula repeated, OR wrap in a subquery:
   ```sql
   SELECT * FROM (
     SELECT e.*, v.name as venue_name, ...
            (...) AS distance_km
     FROM events e JOIN venues v ON ...
     WHERE e.event_date = CURRENT_DATE AND e.is_cancelled = FALSE AND v.latitude IS NOT NULL
   ) sub
   WHERE distance_km <= $3
   ORDER BY distance_km ASC
   LIMIT $4
   ```
3. Call `mapDbEventsWithHeadliner` to hydrate lineup data for each event, then manually add `distanceKm` from the subquery result to each event object.
4. Return type: `Promise<(Event & { distanceKm: number })[]>`
