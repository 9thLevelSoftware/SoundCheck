# Phase 11.2: Mobile Review Response UI - Research

**Researched:** 2026-02-28
**Domain:** Flutter mobile UI + backend review response integration
**Confidence:** HIGH

## Summary

Phase 11.2 closes the audit gap for VERIFY-05: the backend endpoint `POST /api/claims/reviews/:reviewId/respond` already exists and works (ClaimController.respondToReview -> ReviewService.respondToReview), but no mobile consumer calls it. The mobile app needs: (1) a way for claimed venue owners to see reviews on their venue, (2) a "Respond" action on each review, (3) a text input UI to compose and submit the response, and (4) display of owner responses in the review thread.

The work is primarily mobile-side Flutter (new repository method, providers, UI widgets) with a small but critical backend fix: the `searchReviews` and `getReviewById` queries do NOT select `owner_response` or `owner_response_at` columns, so even though the mapper handles them, the data is never returned to clients. This must be fixed for responses to display.

**Primary recommendation:** Fix backend review queries to include owner_response columns, update mobile Review model to include ownerResponse fields, add respondToReview method to ClaimRepository, build a bottom sheet for response input following the ReportBottomSheet pattern, and add a reviews section to VenueDetailScreen for claimed owners.

### Anti-Patterns to Avoid
- **Using @riverpod codegen for feature providers:** Project consistently uses manual providers for feature code. Only core providers (DioClient, AuthState, repositories) use @riverpod.
- **Creating a separate ReviewRepository:** The respond-to-review endpoint lives under `/claims/` routes, so the method belongs in ClaimRepository, not a new ReviewRepository. For fetching reviews by venue, use DioClient directly in a provider or add to ClaimRepository.
- **Navigating to a new screen for response input:** A bottom sheet is the correct UX pattern (matches report flow). No new route needed.

## Common Pitfalls

### Pitfall 1: Backend searchReviews Missing owner_response Columns
**What goes wrong:** Owner responses are saved but never returned in review lists because the SQL SELECT doesn't include `owner_response` or `owner_response_at`.
**Why it happens:** Migration 036 added columns to the reviews table, but the searchReviews query (lines 266-271) and getReviewById query (lines 100-103) were not updated to select those columns. The mapper (line 564) already handles them, so the data silently comes back as `undefined`.
**How to avoid:** Add `r.owner_response, r.owner_response_at` to ALL review SELECT queries in ReviewService.ts.
**Warning signs:** Reviews display correctly but owner responses never appear even after being submitted.

### Pitfall 2: Freezed Model Regeneration
**What goes wrong:** Adding fields to Review freezed model without running build_runner causes compile errors.
**Why it happens:** freezed requires code generation for `.freezed.dart` and `.g.dart` files.
**How to avoid:** After modifying `review.dart`, run `dart run build_runner build --delete-conflicting-outputs` from the mobile directory.
**Warning signs:** Build fails with "The method 'copyWith' isn't defined" or missing factory constructor errors.

### Pitfall 3: Review Model Field Naming Mismatch
**What goes wrong:** Backend returns `ownerResponse` (camelCase from the mapper), but freezed might expect snake_case or vice versa.
**Why it happens:** Backend mapper (ReviewService.ts line 564) converts to camelCase. Mobile Review model uses freezed with default camelCase field names.
**How to avoid:** Verify the JSON key names match. The backend returns `ownerResponse` and `ownerResponseAt` (camelCase), which matches Dart conventions. If the model uses freezed with default naming, it works. If @JsonKey overrides exist, check them. The existing VerificationClaim.fromJson (line 27-39 of claim_repository.dart) supports both cases -- consider same pattern for Review if needed.

### Pitfall 4: Venue Detail Screen Doesn't Show Reviews
**What goes wrong:** Currently VenueDetailScreen has no reviews section -- it shows check-ins, events, and insights but not reviews. Owner can't see reviews to respond to.
**Why it happens:** Reviews as a feature were built as a standalone domain but never integrated into venue/band detail screens.
**How to avoid:** Add a reviews section to VenueDetailScreen (for claimed owners). Fetch reviews via GET /api/reviews/venue/:venueId and display them with a "Respond" action.

### Pitfall 5: Optimistic UI vs. Reload After Submit
**What goes wrong:** After submitting a response, the review list doesn't update to show the response.
**Why it happens:** FutureProvider.autoDispose needs to be invalidated after mutation.
**How to avoid:** After successful response submission, call `ref.invalidate(venueReviewsProvider(venueId))` to refetch the review list.

## Open Questions

1. **Review model field mapping: `text` vs `content`**
   - What we know: The mobile Review model has a `text` field, but the backend returns `content`. The freezed model may need a `@JsonKey(name: 'content')` annotation on the `text` field, or the field should be renamed to `content` for consistency.
   - What's unclear: Whether the existing `.g.dart` already handles this mapping (it might already have the annotation since the model works somewhere).
   - Recommendation: Check the generated `.g.dart` file or align field names. Using `content` to match the backend is cleaner.

2. **Should band detail also get review responses?**
   - What we know: VERIFY-05 says "Claimed venue owner can view aggregate ratings and respond to reviews." The respondToReview endpoint works for both venues AND bands (ReviewService checks both venueId and bandId ownership).
   - What's unclear: Whether the scope of 11.2 includes band owner responses or only venue owners.
   - Recommendation: Build the response UI generically so it works for both, but the venue detail screen is the primary target for this phase per the requirement text.
