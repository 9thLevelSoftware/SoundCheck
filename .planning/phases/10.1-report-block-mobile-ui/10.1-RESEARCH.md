# Phase 10.1: Report & Block Mobile UI - Research

**Researched:** 2026-02-27
**Domain:** Flutter mobile UI for trust & safety interactions (report content, block users)
**Confidence:** HIGH

## Summary

Phase 10.1 closes an audit gap: the backend report and block APIs (built in Phase 9) have no mobile entry points. The backend is fully operational -- `POST /api/reports` accepts content reports with type/reason/description, and `POST/DELETE /api/blocks/:userId/block` plus `GET /api/blocks/:userId/status` handle block/unblock/status checks. No new backend work is needed.

The mobile work is a pure Flutter UI layer: two new repositories (ReportRepository, BlockRepository) following the established DioClient pattern, Riverpod providers, and UI integration points across three surfaces: feed cards (report via overflow menu), check-in detail screen (report check-ins/comments/photos + report button in app bar), and user profile screen (block/unblock button). The user profile screen (`UserProfileScreen`) is currently a stub -- it shows only a user ID -- so it needs fleshing out to host the block button.

**Primary recommendation:** Create a `reporting` feature directory with `ReportRepository` and a reusable `ReportBottomSheet` widget, plus add `BlockRepository` and block UI to the existing `profile` feature. No new packages needed -- everything uses existing Dio, Riverpod, and Flutter Material patterns.

### Pattern 1: Repository Pattern (DioClient)
**What:** All API calls go through a Repository class that depends on DioClient
**When to use:** Every new API endpoint
**Example:**
```dart
// Source: existing rsvp_repository.dart pattern
class ReportRepository {
  final DioClient _dioClient;
  ReportRepository({required DioClient dioClient}) : _dioClient = dioClient;

  Future<void> submitReport({
    required String contentType,
    required String contentId,
    required String reason,
    String? description,
  }) async {
    await _dioClient.post('/reports', data: {
      'contentType': contentType,
      'contentId': contentId,
      'reason': reason,
      if (description != null) 'description': description,
    });
  }
}
```

### Pattern 2: Manual Riverpod Providers (not codegen)
**What:** Simple features use `Provider`, `FutureProvider`, and `StateNotifierProvider` directly rather than `@riverpod` codegen
**When to use:** Phase 10 decision [10-05]: "Manual Riverpod providers (not @riverpod code-gen) for simple cases matching detail screen patterns"
**Example:**
```dart
// Source: existing event_providers.dart
final reportRepositoryProvider = Provider<ReportRepository>((ref) {
  final dioClient = ref.watch(dioClientProvider);
  return ReportRepository(dioClient: dioClient);
});

final blockStatusProvider = FutureProvider.autoDispose.family<bool, String>(
  (ref, userId) async {
    return ref.watch(blockRepositoryProvider).isBlocked(userId);
  },
);
```

### Pattern 3: Bottom Sheet for Multi-Step Actions
**What:** Slide-up bottom sheets for actions requiring user input (reason selection, optional description)
**When to use:** Report submission flow
**Example reference:** `rating_bottom_sheet.dart`, `photo_upload_sheet.dart`

**What:** AlertDialog with clear warning text for irreversible or high-impact actions

### Anti-Patterns to Avoid
- **Reporting your own content:** The backend doesn't prevent this, but the mobile UI should hide the report option on your own check-ins/comments. Check `userId` against `authState.value?.id`.
- **Blocking yourself:** The backend returns 400 for self-block, but never show the block button on your own profile. The router already has separate `/profile` (self) vs `/users/:id` (others) routes.
- **Stale block status after toggle:** After a block/unblock action, invalidate the relevant providers (`blockStatusProvider`, feed providers) to reflect the new state immediately.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Report content type enum | Custom validation | Match backend enum exactly: `checkin`, `comment`, `photo`, `user` | Backend Zod schema rejects unknown types |
| Report reason enum | Custom enum | Match backend enum exactly: `spam`, `harassment`, `inappropriate`, `copyright`, `other` | Backend Zod validation |
| Bilateral block filtering | Client-side content filtering | Backend already filters bilaterally (getBlockFilterSQL in BlockService) | After blocking, server-side queries already exclude blocked user content from feed/comments |
| Rate limiting UI feedback | Custom throttle logic | Surface backend 429 response message ("Report limit reached...") | Backend rate limits: 10 reports/day, 30 block actions/15min |
| Optimistic block toggle | Complex local state sync | Simple invalidate-and-refetch after block/unblock | Block is infrequent; no need for optimistic updates |

**Key insight:** The backend already handles all validation, rate limiting, deduplication (UNIQUE constraint on reports), and bilateral content filtering. The mobile layer just needs to surface the UI and forward requests.

## Common Pitfalls

### Pitfall 1: Using Wrong Content ID for Photo Reports
**What goes wrong:** Photos are stored as `image_urls` array on the check-ins table, not as separate entities with their own IDs
**Why it happens:** Developer assumes photos have their own UUIDs
**How to avoid:** Per Phase 9 decision [09-02]: "Photo reports use checkin ID as contentId (photos stored as image_urls on checkins table)". Use the check-in ID as `contentId` with `contentType: 'photo'`.
**Warning signs:** 404 errors from report API when trying to report a photo

### Pitfall 2: Not Prefixing API Paths
**What goes wrong:** DioClient's baseUrl already includes `/api` suffix (e.g., `https://soundcheck-app.up.railway.app/api`). Passing `/api/reports` would double the prefix.
**Why it happens:** Looking at backend route mounting (`app.use('/api/reports', ...)`) and using the full path
**How to avoid:** Repository paths should be WITHOUT the `/api` prefix: use `/reports`, `/blocks/:userId/block`, etc. DioClient handles the base URL.
**Warning signs:** 404 errors from all API calls

### Pitfall 3: Forgetting to Invalidate Feed After Block
**What goes wrong:** User blocks someone but still sees their content in the feed until manual refresh
**Why it happens:** Feed data is cached by Riverpod providers
**How to avoid:** After successful block, invalidate feed providers and any visible content providers
**Warning signs:** Blocked user's content still visible in feed

### Pitfall 4: UserProfileScreen Is a Stub
**What goes wrong:** Developer assumes UserProfileScreen has a full UI and tries to "just add a block button"
**Why it happens:** It was scaffolded as a placeholder
**How to avoid:** Plan includes fleshing out UserProfileScreen with actual user data (avatar, username, stats) before adding the block button. Needs a provider to fetch user profile data.
**Warning signs:** Block button renders on a nearly empty screen

### Pitfall 5: Missing Error Handling for 429 Rate Limits
**What goes wrong:** User hits report rate limit (10/day) or block rate limit (30/15min) and sees generic error
**Why it happens:** DioClient maps 429 to generic ServerFailure
**How to avoid:** Either add 429 handling to DioClient or handle the specific error message in the report/block UI. The backend returns descriptive messages.
**Warning signs:** "Request failed" error when user has been actively reporting

### Block Repository
```dart
// Follows: rsvp_repository.dart pattern
import '../../../core/api/dio_client.dart';

class BlockRepository {
  final DioClient _dioClient;
  BlockRepository({required DioClient dioClient}) : _dioClient = dioClient;

  /// Block a user. POST /blocks/:userId/block
  Future<void> blockUser(String userId) async {
    await _dioClient.post('/blocks/$userId/block');
  }

  /// Unblock a user. DELETE /blocks/:userId/block
  Future<void> unblockUser(String userId) async {
    await _dioClient.delete('/blocks/$userId/block');
  }

  /// Check block status. GET /blocks/:userId/status
  /// Returns { blocked: boolean }
  Future<bool> isBlocked(String userId) async {
    final response = await _dioClient.get('/blocks/$userId/status');
    return response.data['data']['blocked'] as bool;
  }

  /// Get all blocked users. GET /blocks
  Future<List<Map<String, dynamic>>> getBlockedUsers() async {
    final response = await _dioClient.get('/blocks');
    return List<Map<String, dynamic>>.from(response.data['data'] as List);
  }
}
```

## Backend API Reference

### Report API
| Method | Path | Auth | Body | Response |
|--------|------|------|------|----------|
| POST | /api/reports | JWT | `{ contentType, contentId, reason, description? }` | `{ success: true, data: report, message: "Report submitted..." }` |

**contentType enum:** `checkin`, `comment`, `photo`, `user`
**reason enum:** `spam`, `harassment`, `inappropriate`, `copyright`, `other`
**Rate limit:** 10 reports/user/day (429 when exceeded)
**Deduplication:** UNIQUE constraint -- same user can't report same content twice

### Block API
| Method | Path | Auth | Body | Response |
|--------|------|------|------|----------|
| POST | /api/blocks/:userId/block | JWT | none | `{ success: true, data: block }` |
| DELETE | /api/blocks/:userId/block | JWT | none | `{ success: true, message: "User unblocked" }` |
| GET | /api/blocks/:userId/status | JWT | none | `{ success: true, data: { blocked: boolean } }` |
| GET | /api/blocks | JWT | none | `{ success: true, data: [blocks] }` |

**Rate limit:** 30 actions/15 minutes
**Side effects:** Blocking auto-unfollows in both directions
**Self-block:** 400 error ("Cannot block yourself")
**Bilateral:** `isBlocked` returns true if EITHER user blocked the other

## Open Questions

1. **UserProfileScreen data source**
   - What we know: The screen currently shows only userId text. It needs real user data (avatar, username, stats).
   - What's unclear: Whether to use the existing `/users/:id` endpoint or concert cred endpoint for profile data
   - Recommendation: Check if `profileRepository.dart` has a `getUserProfile(userId)` method; if not, add one. The existing `GET /api/users/:id` endpoint likely returns user data.

2. **Report confirmation UX**
   - What we know: Success criteria #4 says "Report confirmation is shown to the user after submission"
   - What's unclear: SnackBar vs. success state in the bottom sheet
   - Recommendation: Close bottom sheet on success, show a SnackBar with the backend message ("Report submitted successfully. Our team will review it shortly."). This matches the app's existing feedback pattern.

3. **DioClient 429 handling**
   - What we know: Current DioClient maps 400 to ValidationFailure, 401 to AuthFailure, but has no special 429 handling
   - What's unclear: Whether to add 429 handling globally or handle it per-feature
   - Recommendation: For now, surface the backend error message as-is (it's user-friendly: "Report limit reached..."). The existing error extraction logic in DioClient already parses the `error` field from the response body.
