**TrendingService.ts:**
Create a service following the singleton Database pattern (`private db = Database.getInstance()`).

Implement `getTrendingNearUser(userId: string, lat: number, lon: number, options?: { radiusKm?: number, days?: number, limit?: number }): Promise<TrendingEvent[]>`:

1. Query upcoming events within `radiusKm` (default 80km) and `days` ahead (default 30)
2. For each event, compute signals via LATERAL joins:
   - `rsvp_count`: COUNT from event_rsvps
   - `checkin_velocity`: COUNT from checkins in last 7 days WHERE is_hidden IS NOT TRUE
   - `friend_signals`: COUNT DISTINCT users who RSVP'd AND are followed by the requesting user (via user_followers)
3. Apply Wilson scoring with weighted composite:
   - `positive = (rsvp_count * 3 + checkin_velocity * 2 + friend_signals * 5)`
   - `total = (rsvp_count + checkin_velocity + friend_signals + 1)` (the +1 prevents div/0)
   - Call `wilson_lower_bound(positive::BIGINT, total::BIGINT)` (the SQL function from migration 033)
4. Apply proximity decay: `trending_score = wilson_result * (1.0 / (1.0 + distance_km / 50.0))`
5. Filter: `event_date >= CURRENT_DATE AND event_date <= CURRENT_DATE + INTERVAL '$days days' AND is_cancelled = FALSE`
6. Order by trending_score DESC, limit to `limit` (default 20)
7. Include venue name, city, state, and lineup bands via LEFT JOIN on event_lineup + bands
8. Use Haversine distance formula in SQL (see EventService.getTrendingNearby for existing pattern)
9. Return mapped TrendingEvent[] array

**IMPORTANT CONTEXT:** The `checkins` table already has `toast_count` and `comment_count` columns with triggers that maintain them (`trigger_update_toast_count`, `trigger_update_comment_count`). The services currently compute these via `COUNT(DISTINCT t.id)` and `COUNT(DISTINCT cm.id)` JOINs. Switch to reading the columns directly.

**FeedService.ts — Two queries to update:**

1. **getFriendsFeed query** (~line 90-130): Currently has:
   ```sql
   LEFT JOIN toasts t ON c.id = t.checkin_id
   LEFT JOIN checkin_comments cm ON c.id = cm.checkin_id
   ...
   COUNT(DISTINCT t.id)::int AS toast_count,
   COUNT(DISTINCT cm.id)::int AS comment_count,
   ```
   Replace with:
   ```sql
   -- Remove the LEFT JOIN toasts and LEFT JOIN checkin_comments lines
   -- Replace COUNT(DISTINCT t.id)::int AS toast_count with: c.toast_count,
   -- Replace COUNT(DISTINCT cm.id)::int AS comment_count with: c.comment_count,
   ```

   KEEP the `EXISTS(SELECT 1 FROM toasts t2 WHERE t2.checkin_id = c.id AND t2.user_id = $1) AS has_user_toasted` subquery — this is per-viewer and cannot be denormalized.

2. **getHappeningNow / getEventFeed query** (~line 170-210): Same pattern — replace COUNT JOINs with column reads, keep has_user_toasted EXISTS subquery, remove GROUP BY.
