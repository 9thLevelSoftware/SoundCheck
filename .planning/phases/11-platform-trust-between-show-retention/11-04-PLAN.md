**ClaimService.ts:**
Create service using Database singleton pattern.

Methods:

1. `submitClaim(userId: string, request: CreateClaimRequest): Promise<VerificationClaim>`
   - Validate entityType is 'venue' or 'band'
   - Verify entity exists: query venues/bands table by entityId (throw 404 if not found)
   - Check user doesn't already have an approved claim for this entity (throw ConflictError if so)
   - INSERT into verification_claims with status='pending'. The partial unique index will enforce one-pending-per-entity at DB level — catch unique violation and throw a descriptive error ("A pending claim already exists for this entity")
   - Return the created claim

2. `getMyClaims(userId: string): Promise<VerificationClaim[]>`
   - SELECT from verification_claims WHERE user_id = $1
   - JOIN entity tables to get entity names
   - ORDER BY created_at DESC

3. `getClaimById(claimId: string): Promise<VerificationClaim>`
   - SELECT with JOINs for entity name, user name/email

4. `getPendingClaims(): Promise<VerificationClaim[]>` (admin)
   - SELECT WHERE status = 'pending' with entity/user JOINs
   - ORDER BY created_at ASC (oldest first for FIFO)

5. `getAllClaims(status?: ClaimStatus): Promise<VerificationClaim[]>` (admin)
   - SELECT with optional status filter, JOINs, ORDER BY created_at DESC

6. `reviewClaim(claimId: string, adminUserId: string, decision: ReviewClaimRequest): Promise<VerificationClaim>` (admin)
   - Use a database transaction (BEGIN...COMMIT):
     a. UPDATE verification_claims SET status = decision.status, reviewed_by = adminUserId, reviewed_at = NOW(), review_notes = decision.reviewNotes WHERE id = claimId AND status = 'pending'
     b. If status is 'approved':
        - Determine table from entity_type ('venue' -> venues, 'band' -> bands)
        - UPDATE {table} SET claimed_by_user_id = claim.user_id WHERE id = claim.entity_id
        - Note: Do NOT set is_verified here — is_verified has existing semantics (organic verification via checkins). Use claimed_by_user_id as the signal for claim verification.
     c. If no rows updated in step (a), throw error ("Claim not found or already reviewed")
   - Return updated claim
   - On error, ROLLBACK
